## Biological Background

### Why whole-genome sequencing for variant calling?

WGS is the reference standard for variant discovery. By sequencing every base
in the genome at sufficient depth, it provides:

- **Uniform genome coverage** — no bias toward expressed genes or restriction
  sites; structural variants, intergenic SNPs, and regulatory variants are all
  captured equally
- **High per-locus depth** — at 15–30× depth, genotype calls at most positions
  are supported by enough reads to distinguish sequencing error from true
  variation with high confidence
- **A ground-truth dataset** — WGS calls on well-characterised accessions (like
  those from the 1001 Genomes Project) can be compared to published SNP truth
  sets, making benchmarking straightforward

The tradeoff is cost: at ~$50–100 per sample for library prep and sequencing at
15–30× depth on a 135 Mb genome, WGS is expensive relative to GBS. For the
five accessions used here, the data already exist and were generated by the 1001
Genomes Consortium at no additional cost to this tutorial.

### The GATK4 GVCF workflow — why joint genotyping?

A naive approach to multi-sample variant calling would call each sample
independently and merge the resulting VCFs. This fails at a critical step:
**distinguishing a truly homozygous-reference genotype from a missing data
point**. If sample A has no reads at a position, a per-sample caller has no
information to report — it simply emits nothing. When you later merge, you
cannot tell whether the absence of a call means "homozygous reference" or
"no data."

The GATK4 GVCF workflow solves this:

```
Per sample (run in parallel):
  HaplotypeCaller → sample.g.vcf.gz
  The GVCF contains reference confidence blocks at non-variant sites,
  recording the evidence (or lack thereof) at every position.

Joint genotyping (run once across all samples):
  GenomicsDBImport → consolidates all GVCFs into an efficient database
  GenotypeGVCFs   → examines all samples simultaneously at every site,
                    producing a single cohort VCF with one column per sample
```

The result is a VCF where every sample has a genotype call at every variant
site — either a true variant genotype, a reference genotype supported by
reads, or a `./. ` missing-data call when coverage is genuinely absent. This
distinction is essential for downstream allele frequency estimation, population
structure analysis, and genomic selection.

### Why we use hard filters (not VQSR) for this cohort

GATK's preferred filtering approach is **Variant Quality Score Recalibration
(VQSR)**, a machine-learning model trained on known-variant truth sets. VQSR
requires a minimum of ~30 WGS samples to build a reliable model. With five
samples, we use **hard filters** — explicit thresholds on variant quality
metrics such as QD, FS, MQ, and SOR. Hard filters are less statistically
sophisticated but completely transparent and appropriate for small cohorts.

## Step 1 — BWA-MEM2 Alignment

BWA-MEM2 aligns each pair of trimmed FASTQ files to the TAIR10 reference,
producing a SAM stream that is immediately sorted to BAM with `samtools sort`.
We pipe alignment directly into sorting to avoid writing a large intermediate
SAM file.

### Read groups — a GATK requirement

GATK4 uses **read group** (`@RG`) information embedded in the BAM header to
identify which sample each read belongs to. This is mandatory — GATK will
refuse to run HaplotypeCaller on a BAM without read groups. The key tags are:

| Tag | Meaning | Value used here |
|---|---|---|
| `ID` | Read group identifier | `{sample}_RG1` |
| `SM` | Sample name — **must match across all tools** | Accession name (e.g. `Vas-0`) |
| `PL` | Sequencing platform | `ILLUMINA` |
| `LB` | Library identifier | `{sample}_lib1` |

```bash
cat > ${VARCALL}/scripts/02a_bwa_align.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=bwa_align
#SBATCH --account=PAS3260
#SBATCH --time=04:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=12
#SBATCH --mem=32G
#SBATCH --array=0-4
#SBATCH --output=/fs/scratch/PAS3260/${user_name}/variant_calling/logs/bwa_align_%A_%a.out
#SBATCH --error=/fs/scratch/PAS3260/${user_name}/variant_calling/logs/bwa_align_%A_%a.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling
THREADS=12

SAMPLES=(Vas-0 Bez-9 Gen-8 Mah-6 Usa-0)
SAMPLE=${SAMPLES[$SLURM_ARRAY_TASK_ID]}

REF=${VARCALL}/00_reference/indexes/bwa/Athaliana_TAIR10.fasta
R1=${VARCALL}/02_qc/fastp/${SAMPLE}_wgs_R1.fastq.gz
R2=${VARCALL}/02_qc/fastp/${SAMPLE}_wgs_R2.fastq.gz
OUT=${VARCALL}/03_wgs/aligned/${SAMPLE}_sorted.bam

echo "=== BWA-MEM2 alignment: ${SAMPLE} ==="
echo "Started: $(date)"

[[ -f ${R1} ]] || { echo "ERROR: R1 not found: ${R1}"; exit 1; }
[[ -f ${R2} ]] || { echo "ERROR: R2 not found: ${R2}"; exit 1; }

# Align and sort in a single pipe
# -R: read group string (required by GATK downstream)
# samtools sort -@ threads for parallel compression
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bwamem2_2.2.3.sif \
  bwa-mem2 mem \
    -t ${THREADS} \
    -R "@RG\tID:${SAMPLE}_RG1\tSM:${SAMPLE}\tPL:ILLUMINA\tLB:${SAMPLE}_lib1" \
    ${REF} \
    ${R1} ${R2} | \
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/samtools_1.23.1.sif \
  samtools sort \
    -@ ${THREADS} \
    -T ${VARCALL}/03_wgs/aligned/${SAMPLE}_tmp \
    -o ${OUT}

# Index the sorted BAM
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/samtools_1.23.1.sif \
  samtools index ${OUT}

echo ""
echo "=== Alignment stats for ${SAMPLE} ==="
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/samtools_1.23.1.sif \
  samtools flagstat ${OUT}

echo ""
echo "Output: ${OUT}"
ls -lh ${OUT}
echo "=== ${SAMPLE} alignment complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02a_bwa_align.sh
```

> **Why pipe BWA-MEM2 into samtools sort?** Writing an uncompressed SAM file
> for a 15–30× WGS sample would consume 30–60 GB of scratch space per sample.
> Piping directly into `samtools sort` keeps peak disk usage to ~3–5 GB per
> sample (the sorted, compressed BAM) with no temporary SAM ever written.

### Expected alignment statistics

After completion, `samtools flagstat` should report approximately:

```
~20–40M + 0 in total (QC-passed reads + QC-failed reads)
~20–40M + 0 primary
0 + 0 secondary
~50K + 0 supplementary          ← chimeric/split reads; normal
0 + 0 duplicates                ← not yet marked
~19–39M + 0 mapped (97–99%)    ← good alignment rate
~20–40M + 0 paired in sequencing
~10–20M + 0 read1
~10–20M + 0 read2
~18–36M + 0 properly paired (90–98%)
```

Alignment rates below 90% warrant investigation — check whether the correct
reference was used and whether the reads contain unexpected contamination.

## Step 2 — Picard MarkDuplicates

PCR duplicates are reads that originate from the same original DNA fragment,
amplified multiple times during library preparation. They inflate apparent read
depth without adding information and cause variant callers to overestimate
confidence at affected positions. Picard MarkDuplicates identifies read pairs
with identical 5' mapping coordinates (a signature of PCR duplication) and
flags them in the BAM — GATK then ignores flagged reads during variant calling.

```bash
cat > ${VARCALL}/scripts/02b_markduplicates.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=markdup
#SBATCH --account=PAS3260
#SBATCH --time=02:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
#SBATCH --mem=32G
#SBATCH --array=0-4
#SBATCH --output=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/markdup_%A_%a.out
#SBATCH --error=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/markdup_%A_%a.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling

SAMPLES=(Vas-0 Bez-9 Gen-8 Mah-6 Usa-0)
SAMPLE=${SAMPLES[$SLURM_ARRAY_TASK_ID]}

IN=${VARCALL}/03_wgs/aligned/${SAMPLE}_sorted.bam
OUT=${VARCALL}/03_wgs/aligned/${SAMPLE}_markdup.bam
METRICS=${VARCALL}/03_wgs/aligned/${SAMPLE}_dup_metrics.txt

echo "=== Picard MarkDuplicates: ${SAMPLE} ==="
echo "Started: $(date)"

apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/picard_3.4.0.sif \
  picard MarkDuplicates \
    -I ${IN} \
    -O ${OUT} \
    -M ${METRICS} \
    --REMOVE_DUPLICATES false \
    --VALIDATION_STRINGENCY SILENT \
    --TMP_DIR ${VARCALL}/03_wgs/aligned/

# REMOVE_DUPLICATES false: mark duplicates but keep them in the BAM
#   GATK ignores marked duplicates during variant calling; keeping them
#   allows you to inspect the BAM and recalculate duplicate rates later
# VALIDATION_STRINGENCY SILENT: suppress non-critical BAM validation warnings

# Index the duplicate-marked BAM
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/samtools_1.23.1.sif \
  samtools index ${OUT}

echo ""
echo "=== Duplicate metrics for ${SAMPLE} ==="
grep -A 2 "ESTIMATED_LIBRARY_SIZE" ${METRICS} | tail -2

echo ""
echo "Output: ${OUT}"
ls -lh ${OUT}
echo "=== ${SAMPLE} MarkDuplicates complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02b_markduplicates.sh
```

### Interpreting the duplicate metrics file

The metrics file contains a summary table. The most informative columns are:

| Column | Meaning | Typical WGS value |
|---|---|---|
| `PERCENT_DUPLICATION` | Fraction of reads flagged as duplicates | 5–25% |
| `ESTIMATED_LIBRARY_SIZE` | Estimated number of unique molecules in the library | >5 million |

Values above 40% duplication indicate low library complexity — the physical
library contained fewer unique molecules than the number of reads generated.
This is unusual for whole-genome libraries prepared from high-quality gDNA
but can occur with low-input or degraded samples.

> **Key point:** Marking duplicates does not remove reads. It sets a flag
> (SAM flag `0x400`) that GATK reads during HaplotypeCaller. The original BAM
> is preserved; only the flag is added.

## Step 3 — GATK4 HaplotypeCaller (per-sample GVCF)

HaplotypeCaller is run independently on each sample, producing a **GVCF**
(Genomic VCF). Unlike a standard VCF, the GVCF records information at every
position in the genome — not just variant sites. Non-variant positions are
compressed into reference blocks (e.g. `<NON_REF>` allele entries covering
ranges of positions with similar confidence). This is what enables the joint
genotyping in Step 4 to distinguish "homozygous reference" from "no data."

```bash
cat > ${VARCALL}/scripts/02c_haplotypecaller.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=haplotypecaller
#SBATCH --account=PAS3260
#SBATCH --time=08:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
#SBATCH --mem=32G
#SBATCH --array=0-4
#SBATCH --output=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/hc_%A_%a.out
#SBATCH --error=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/hc_%A_%a.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling

SAMPLES=(Vas-0 Bez-9 Gen-8 Mah-6 Usa-0)
SAMPLE=${SAMPLES[$SLURM_ARRAY_TASK_ID]}

REF=${VARCALL}/00_reference/fasta/Athaliana_TAIR10.fasta
BAM=${VARCALL}/03_wgs/aligned/${SAMPLE}_markdup.bam
GVCF=${VARCALL}/03_wgs/gvcf/${SAMPLE}.g.vcf.gz

echo "=== HaplotypeCaller: ${SAMPLE} ==="
echo "Started: $(date)"

apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk --java-options "-Xmx24g -Xms4g" HaplotypeCaller \
    --reference ${REF} \
    --input ${BAM} \
    --output ${GVCF} \
    --emit-ref-confidence GVCF \
    --sample-name ${SAMPLE} \
    --pcr-indel-model NONE \
    --native-pair-hmm-threads 4

# --emit-ref-confidence GVCF: produce GVCF instead of standard VCF
# --sample-name: explicitly set the sample name in the output
#   (must match the SM: tag in the BAM read group)
# --pcr-indel-model NONE: appropriate for PCR-free or high-quality libraries;
#   1001 Genomes libraries were standard PCR-based — use CONSERVATIVE if
#   you observe elevated false-positive indels
# --native-pair-hmm-threads: use multiple threads for the pair-HMM step
#   (the most compute-intensive part of HaplotypeCaller)
# -Xmx24g: maximum Java heap = 24 GB (80% of the 32 GB SLURM allocation)

echo ""
echo "GVCF output:"
ls -lh ${GVCF}

# Quick check: count variant sites in the GVCF
echo ""
echo "Variant records in GVCF (non-reference blocks):"
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools view -v snps,indels ${GVCF} | \
  bcftools stats | grep "^SN" | grep "number of SNPs"

echo "=== ${SAMPLE} HaplotypeCaller complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02c_haplotypecaller.sh
```

> ⚠️ **HaplotypeCaller is the most time-consuming step in this module.**
> Each sample takes approximately 2–6 hours on a single node depending on
> genome coverage and node load. The SLURM array runs all five samples
> simultaneously — total wall time equals the slowest single sample, not
> the sum of all five.

## Step 4 — GenomicsDBImport

GenomicsDBImport consolidates the five per-sample GVCFs into a single
efficient database structure (a GenomicsDB workspace). This step is required
before joint genotyping and is run **once across all samples together**.

We process each chromosome as a separate interval to keep memory usage
manageable. Here we combine all five nuclear chromosomes in a single job,
which is appropriate for the 135 Mb *Arabidopsis* genome.

```bash
cat > ${VARCALL}/scripts/02d_genomicsdb.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=genomicsdb
#SBATCH --account=PAS3260
#SBATCH --time=04:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
#SBATCH --mem=32G
#SBATCH --output=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/genomicsdb_%j.out
#SBATCH --error=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/genomicsdb_%j.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling
DB=${VARCALL}/03_wgs/genotyped/genomicsdb

echo "=== GenomicsDBImport ==="
echo "Started: $(date)"

# Build the -V sample input arguments
SAMPLE_ARGS=""
for SAMPLE in Vas-0 Bez-9 Gen-8 Mah-6 Usa-0; do
    GVCF=${VARCALL}/03_wgs/gvcf/${SAMPLE}.g.vcf.gz
    [[ -f ${GVCF} ]] || { echo "ERROR: GVCF not found: ${GVCF}"; exit 1; }
    SAMPLE_ARGS="${SAMPLE_ARGS} -V ${GVCF}"
done

echo "Sample GVCFs:"
for SAMPLE in Vas-0 Bez-9 Gen-8 Mah-6 Usa-0; do
    ls -lh ${VARCALL}/03_wgs/gvcf/${SAMPLE}.g.vcf.gz
done

apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk --java-options "-Xmx24g -Xms4g" GenomicsDBImport \
    ${SAMPLE_ARGS} \
    --genomicsdb-workspace-path ${DB} \
    --intervals 1 \
    --intervals 2 \
    --intervals 3 \
    --intervals 4 \
    --intervals 5 \
    --reader-threads 4 \
    --batch-size 50 \
    --merge-input-intervals

# --intervals: process all 5 nuclear chromosomes
#   Mt and Pt are excluded: organellar genomes have distinct
#   evolutionary pressures and are typically analysed separately
# --batch-size 50: number of samples to process per batch
#   (irrelevant for 5 samples; important for 100+ sample cohorts)
# --merge-input-intervals: combine all interval records into one DB
#   (required when specifying multiple --intervals flags)

echo ""
echo "GenomicsDB workspace:"
ls -lh ${DB}/
echo "=== GenomicsDBImport complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02d_genomicsdb.sh
```

> ⚠️ **GenomicsDBImport cannot be restarted mid-run.** If the job fails,
> you must delete the workspace directory and restart from scratch:
> ```bash
> rm -rf ${VARCALL}/03_wgs/genotyped/genomicsdb
> ```
> Always check that all five GVCFs exist and are non-empty before submitting.

## Step 5 — GenotypeGVCFs (Joint Genotyping)

GenotypeGVCFs reads from the GenomicsDB workspace and produces a standard
multi-sample VCF with one column per accession. At each variant site, every
sample receives a genotype call based on the joint likelihood of all observed
reads across the cohort.

```bash
cat > ${VARCALL}/scripts/02e_genotypegvcfs.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=genotypegvcfs
#SBATCH --account=PAS3260
#SBATCH --time=02:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
#SBATCH --mem=32G
#SBATCH --output=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/genotypegvcfs_%j.out
#SBATCH --error=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/genotypegvcfs_%j.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling

REF=${VARCALL}/00_reference/fasta/Athaliana_TAIR10.fasta
DB=${VARCALL}/03_wgs/genotyped/genomicsdb
RAW_VCF=${VARCALL}/03_wgs/genotyped/cohort_raw.vcf.gz

echo "=== GenotypeGVCFs ==="
echo "Started: $(date)"

# Verify the GenomicsDB workspace exists before proceeding
[[ -d ${DB} ]] || { echo "ERROR: GenomicsDB workspace not found: ${DB}"; exit 1; }
echo "GenomicsDB workspace confirmed: ${DB}"

apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk --java-options "-Xmx24g -Xms4g" GenotypeGVCFs \
    --reference ${REF} \
    --variant gendb://${DB} \
    --output ${RAW_VCF} \
    --intervals 1 \
    --intervals 2 \
    --intervals 3 \
    --intervals 4 \
    --intervals 5 \
    --merge-input-intervals \
    --include-non-variant-sites false

# --merge-input-intervals: required when multiple --intervals flags are used,
#   to match the way GenomicsDBImport consolidated the chromosomes in 02d
# Removed --standard-min-confidence-threshold-for-calling: deprecated in
#   GATK 4.3+; GATK 4.6.2.0 uses a default calling threshold internally

echo ""
echo "Raw cohort VCF:"
ls -lh ${RAW_VCF}

# Index the raw cohort VCF — required by GATK SelectVariants in 02f
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk IndexFeatureFile \
    -I ${RAW_VCF}

echo ""
echo "Index created: ${RAW_VCF}.tbi"

echo ""
echo "=== Basic site counts ==="
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools stats ${RAW_VCF} | grep "^SN"

echo "=== GenotypeGVCFs complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02e_genotypegvcfs.sh
```

## Step 6 — Hard Filtering

The raw VCF from GenotypeGVCFs contains many low-confidence variant calls
that must be filtered before downstream analysis. With only five samples,
VQSR is not applicable — we instead apply hard thresholds to INFO-level
annotation fields that capture different aspects of call quality.

We filter SNPs and indels separately because they have different error
profiles and require different thresholds.

### Understanding the filter annotations

| Annotation | Meaning | Why it matters |
|---|---|---|
| `QD` | Quality by Depth: variant quality (QUAL) divided by total depth | Low QD indicates the QUAL score is inflated by high depth alone |
| `FS` | Fisher Strand: Phred-scaled p-value of strand bias | High FS means the variant is seen predominantly on one strand — a sequencing artefact signature |
| `SOR` | Strand Odds Ratio: another strand bias metric | More robust than FS at high-coverage sites |
| `MQ` | Root mean square mapping quality of reads at the site | Low MQ indicates reads supporting the variant align poorly, suggesting a repeat region |
| `MQRankSum` | Difference in mapping quality between ref and alt-supporting reads | Large negative values suggest alt reads align worse — mapping artefact |
| `ReadPosRankSum` | Difference in read position between ref and alt-supporting reads | Large negative values suggest the alt allele is predominantly at read ends — sequencing artefact |

```bash
cat > ${VARCALL}/scripts/02f_hard_filter.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=hard_filter
#SBATCH --account=PAS3260
#SBATCH --time=01:00:00
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=2
#SBATCH --mem=16G
#SBATCH --output=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/hard_filter_%j.out
#SBATCH --error=/fs/scratch/PAS3260/Jonathan/variant_calling/logs/hard_filter_%j.err

set -euo pipefail

# ↓ Replace <your_username> with your OSC username (e.g. jsmith)
user_name=<your_username>
VARCALL=/fs/scratch/PAS3260/${user_name}/variant_calling
REF=${VARCALL}/00_reference/fasta/Athaliana_TAIR10.fasta

RAW_VCF=${VARCALL}/03_wgs/genotyped/cohort_raw.vcf.gz
FILT_DIR=${VARCALL}/03_wgs/filtered

echo "=== Hard filtering: $(date) ==="

# ---- Step A: Separate SNPs and indels ----
echo "[A] Selecting SNPs..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk SelectVariants \
    -R ${REF} \
    -V ${RAW_VCF} \
    --select-type-to-include SNP \
    -O ${FILT_DIR}/snps_raw.vcf.gz

echo "[A] Selecting indels..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk SelectVariants \
    -R ${REF} \
    -V ${RAW_VCF} \
    --select-type-to-include INDEL \
    -O ${FILT_DIR}/indels_raw.vcf.gz

# ---- Step B: Apply SNP hard filters ----
echo "[B] Applying SNP hard filters..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk VariantFiltration \
    -R ${REF} \
    -V ${FILT_DIR}/snps_raw.vcf.gz \
    --filter-expression "QD < 2.0"           --filter-name "QD2" \
    --filter-expression "FS > 60.0"          --filter-name "FS60" \
    --filter-expression "SOR > 3.0"          --filter-name "SOR3" \
    --filter-expression "MQ < 40.0"          --filter-name "MQ40" \
    --filter-expression "MQRankSum < -12.5"  --filter-name "MQRankSum-12.5" \
    --filter-expression "ReadPosRankSum < -8.0" --filter-name "ReadPosRankSum-8" \
    -O ${FILT_DIR}/snps_filtered.vcf.gz

# ---- Step C: Apply indel hard filters ----
echo "[C] Applying indel hard filters..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk VariantFiltration \
    -R ${REF} \
    -V ${FILT_DIR}/indels_raw.vcf.gz \
    --filter-expression "QD < 2.0"              --filter-name "QD2" \
    --filter-expression "FS > 200.0"            --filter-name "FS200" \
    --filter-expression "SOR > 10.0"            --filter-name "SOR10" \
    --filter-expression "ReadPosRankSum < -20.0" --filter-name "ReadPosRankSum-20" \
    -O ${FILT_DIR}/indels_filtered.vcf.gz

# ---- Step D: Merge filtered SNPs and indels ----
echo "[D] Merging filtered SNPs and indels..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/gatk4_4.6.2.0.sif \
  gatk MergeVcfs \
    -I ${FILT_DIR}/snps_filtered.vcf.gz \
    -I ${FILT_DIR}/indels_filtered.vcf.gz \
    -O ${FILT_DIR}/cohort_filtered.vcf.gz

# ---- Step E: Extract PASS sites only ----
echo "[E] Extracting PASS sites..."
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools view \
    -f PASS \
    ${FILT_DIR}/cohort_filtered.vcf.gz \
    -O z -o ${FILT_DIR}/cohort_PASS.vcf.gz

apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools index --tbi ${FILT_DIR}/cohort_PASS.vcf.gz

echo ""
echo "=== Filtering summary ==="
echo "Raw variants:"
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools stats ${RAW_VCF} | grep "^SN" | grep -E "SNPs|indels"

echo ""
echo "PASS variants (after filtering):"
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools stats ${FILT_DIR}/cohort_PASS.vcf.gz | grep "^SN" | grep -E "SNPs|indels"

echo "=== Hard filtering complete: $(date) ==="
EOF

sbatch ${VARCALL}/scripts/02f_hard_filter.sh
```

> **VariantFiltration marks, not removes.** After Step B and C, the FILTER
> column contains either `PASS` or the name(s) of failed filters (e.g.
> `FS60;MQ40`). Step E uses `bcftools view -f PASS` to extract only clean
> sites into the final `cohort_PASS.vcf.gz`.

### 7.2 What to expect from the final VCF

For five *Arabidopsis* accessions sequenced at 14–32× depth, the `cohort_PASS.vcf.gz`
should contain approximately:

| Metric | Expected range | Notes |
|---|---|---|
| Total SNPs | 700,000–2,000,000 | Depends on genetic distance between accessions |
| Total indels | 50,000–200,000 | Typically ~10% of SNP count |
| Ts/Tv ratio | 2.0–2.3 | A ratio outside 1.8–2.5 suggests filtering issues |
| Missing genotypes per sample | < 5% | Higher in Gen-8 due to lower coverage |
| Mean site depth | 15–30× | Reflects the 14–32× coverage range across samples |

> **Ts/Tv ratio as a quality indicator.** Transitions (Ts: A↔G, C↔T) are
> chemically more likely than transversions (Tv: A↔C, A↔T, G↔C, G↔T) at
> genuine variant sites. A Ts/Tv ratio well below 2.0 suggests that many
> remaining "variants" are actually sequencing errors (which do not follow
> the biological Ts/Tv expectation). A ratio above 2.5 is unusual but
> can occur in GC-rich or CpG-enriched sequence contexts.

## Step 8 — Inspect the VCF Format

Understanding the VCF format is essential before moving to annotation and
filtering in Module 05. Examine a few records from your final VCF:

```bash
# Print VCF header
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \

# Print the first 5 SNP records
apptainer exec \
  --bind ${VARCALL}:${VARCALL} \
  ${VARCALL}/containers/bcftools_1.23.1.sif \
  bcftools view -v snps ${VARCALL}/03_wgs/filtered/cohort_PASS.vcf.gz | \
  grep -v "^#" | head -5
```

A typical SNP record looks like this (column names added for clarity):

```
CHROM  POS     ID  REF  ALT  QUAL   FILTER  INFO              FORMAT    Vas-0      Bez-9
1      142387  .   G    A    543.2  PASS    AC=3;AF=0.3;AN=10  GT:AD:DP:GQ  0/0:22,0:22:60  1/1:0,19:19:57
```

Key fields to understand:

| Field | Value | Meaning |
|---|---|---|
| `CHROM` | `1` | Chromosome 1 |
| `POS` | `142387` | Position on the chromosome (1-based) |
| `REF` | `G` | Reference allele (from TAIR10 Col-0) |
| `ALT` | `A` | Alternate allele found in one or more samples |
| `QUAL` | `543.2` | Phred-scaled quality of the ALT call |
| `FILTER` | `PASS` | All hard filters passed |
| `AC` | `3` | Alternate allele count across all samples |
| `AF` | `0.3` | Alternate allele frequency in this cohort |
| `AN` | `10` | Total allele count (5 diploid samples × 2) |
| `GT` | `0/0` / `1/1` | Genotype: 0=REF, 1=ALT |
| `AD` | `22,0` / `0,19` | Allele depth: ref reads, alt reads |
| `DP` | `22` / `19` | Total depth at this site for this sample |
| `GQ` | `60` / `57` | Genotype quality (Phred-scaled confidence) |
